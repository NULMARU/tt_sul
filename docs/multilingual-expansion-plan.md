# 다국어 학습 확장 검토 및 개발 순서

작성일: 2026-05-17  
기준 프로젝트: `/Users/spacenulmaru/Projects/projects/sulsul-plus`

## 1. 결론

현재 앱의 학습 포맷은 영어 전용 앱으로 시작했지만, 구조 자체는 다른 언어 학습으로 확장 가능하다. 핵심 루프인 `표현 → 듣기/읽기 → 출력 → 퀴즈 → 낙서장 → 오답복습 → 레벨테스트`는 베트남어와 일본어에도 적용할 수 있다.

다만 현재 코드와 콘텐츠에는 영어 전용 전제가 많다. 따라서 영어 개발을 먼저 완료하고 충분히 테스트·개선한 뒤, 같은 엔진을 언어별 설정과 콘텐츠팩으로 확장한다.

개발 순서는 다음으로 확정한다.

| 순서 | 언어 | 상태 | 목표 |
|---|---|---|---|
| 1 | 영어 | 현재 개발 중 | 초급/중급/상급, 승급 미션, 낙서장 인사이트까지 안정화 |
| 2 | 베트남어 | 다음 확장 | 성조, 발음, 생활회화 중심으로 첫 다국어 검증 |
| 3 | 일본어 | 세 번째 확장 | 문자 체계, 후리가나, 존댓말까지 반영 |

## 2. 같은 포맷으로 확장 가능한 이유

| 현재 앱 포맷 | 다국어 적용 |
|---|---|
| Phrase 카드 | 언어별 핵심 표현 카드로 재사용 |
| Daily Story | 언어별 짧은 이야기 또는 대화문으로 재사용 |
| 6종 퀴즈 | 언어별 정답 판정 규칙만 바꾸면 재사용 가능 |
| Journal/낙서장 | 한국어 → 목표언어 작문/번역/문장검사로 확장 |
| Review/SRS | 언어와 무관하게 재사용 가능 |
| 승급 미션 | 언어별 레벨테스트/승급시험으로 재사용 |
| Content Lab | 언어별 약점 보강 콘텐츠 생성으로 확장 |

## 3. 영어 전용 전제와 수정 필요 영역

| 영역 | 현재 상태 | 확장 시 필요 작업 |
|---|---|---|
| 데이터 필드 | `en`, `ko` 중심 | `languageId`, `romanization`, 언어별 표기 필드 추가 |
| TTS/STT | `en-US`, `ko-KR` 중심 | `vi-VN`, `ja-JP` 등 언어별 locale 설정 |
| 퀴즈 정답 판정 | 영어 소문자/공백 normalize | 베트남어 성조, 일본어 표기 변형 허용 정책 |
| 어순 배열 | 영어 공백 단어 기준 | 일본어는 그대로 쓰기 어려움. 문절/패턴 배열로 별도 설계 |
| 낙서장 번역 | 한국어 → 영어 | 한국어 → 목표언어로 일반화 |
| 작문 검사 | 영어 문법 검사 중심 | 언어별 LLM 프롬프트와 평가 기준 필요 |
| 콘텐츠팩 | 영어 seed 파일 | 언어별 seed 파일 또는 콘텐츠팩 구조 필요 |

## 4. 베트남어 확장 시 고려사항

베트남어는 영어 포맷과 비교적 잘 맞는다. 띄어쓰기가 있고 A/B 대화, 역할 응답, 낙서장 작문 흐름을 그대로 적용하기 쉽다.

필수 고려사항:

| 항목 | 설명 |
|---|---|
| 성조 | 정답 판정에서 부호 포함/미포함 허용 정책을 정해야 함 |
| 발음 | TTS/STT locale은 `vi-VN` 기준으로 시작 |
| 로마자 | 기본 문자가 라틴 기반이므로 별도 후리가나는 필요 없지만 발음 힌트는 유용 |
| 콘텐츠 | 여행, 식당, 인사, 가족, 시장, 교통 중심 초급팩부터 시작 |
| 지역차 | 북부/남부 표현 차이는 초급 이후에 다룬다 |

권장 첫 베트남어 범위:

| 단계 | 내용 |
|---|---|
| Stage 1 | 인사, 자기소개, 숫자, 음식 주문, 택시/길 묻기 |
| Stage 2 | A/B 대화 암송, 시장/식당/호텔/친구 대화 |
| Stage 3 | 짧은 의견, 여행 후기, 문화 비교 |

## 5. 일본어 확장 시 고려사항

일본어는 문자와 문법 구조가 영어/베트남어와 달라 추가 추상화가 더 필요하다.

필수 고려사항:

| 항목 | 설명 |
|---|---|
| 문자 체계 | 히라가나, 가타카나, 한자, 후리가나 표시 필요 |
| 정답 판정 | 한자/가나 표기 차이, 띄어쓰기 없음, 조사 오류를 고려 |
| 어순 퀴즈 | 영어식 단어 배열보다 문절 배열 또는 패턴 완성형이 적합 |
| 존댓말 | 보통체/정중체를 콘텐츠 난이도와 상황에 반영 |
| 발음 | `ja-JP` TTS/STT와 음절 단위 쉐도잉 필요 |

권장 첫 일본어 범위:

| 단계 | 내용 |
|---|---|
| Stage 1 | 인사, 자기소개, 기본 동작, 음식 주문 |
| Stage 2 | 정중체 대화, 길 묻기, 쇼핑, 약속 잡기 |
| Stage 3 | 의견 말하기, 문화/업무 대화, 요약 |

## 6. IndexedDB 사용 검토

질문에서 말한 `index.db`는 현재 앱 기준으로는 브라우저 `IndexedDB`로 이해한다. 현재 앱은 `Zustand persist + idb-keyval`을 통해 사용자 데이터를 IndexedDB에 저장한다.

결론은 다음과 같다.

| 질문 | 판단 |
|---|---|
| 영어 단일 앱에서 IndexedDB를 계속 써도 되는가? | 문제 없음 |
| 베트남어/일본어까지 로컬 단일 기기에서 쓸 수 있는가? | 가능 |
| 언어별 데이터가 섞일 위험은? | 있음. 언어별 저장 키 분리 필요 |
| 계정/동기화/운영자 통계까지 가능한가? | IndexedDB만으로는 부족 |

언어별 IndexedDB 저장 키는 다음 규칙을 사용한다.

| 언어 | 저장 키 |
|---|---|
| 영어 | `sulsul-plus:user-state` |
| 베트남어 | `sulsul-plus:user-state:vi` |
| 일본어 | `sulsul-plus:user-state:ja` |

영어는 기존 사용자 데이터 보호를 위해 legacy key를 유지한다. 새 언어부터 언어별 suffix를 붙인다.

## 7. 중앙 DB가 필요한 시점

초기에는 IndexedDB로 충분하다. 하지만 다음 요구가 생기면 서버 DB를 도입해야 한다.

| 요구 | IndexedDB 한계 |
|---|---|
| 기기 간 동기화 | 브라우저 로컬 저장이라 동기화 불가 |
| 로그인 계정 | 사용자별 서버 저장소 필요 |
| 운영자용 콘텐츠 관리 | 정적 seed 파일만으로는 관리 어려움 |
| 학습 통계 분석 | 사용자 동의 기반 중앙 집계 필요 |
| 결제/구독 | 서버 DB와 인증 필요 |
| 콘텐츠 버전 관리 | 언어별 콘텐츠 배포/승인 워크플로 필요 |

장기 권장 구조:

| 계층 | 역할 |
|---|---|
| IndexedDB | 오프라인 학습 상태, 임시 캐시, 빠른 복습 |
| 서버 DB | 계정, 백업, 동기화, 콘텐츠 버전, 운영 통계 |
| 정적 seed | 기본 콘텐츠 번들 |
| Cloudflare KV | LLM 캐시, rate limit |

## 8. 구현 반영 사항

1차로 다음 기반을 코드에 반영했다.

| 파일 | 내용 |
|---|---|
| `src/types/schema.ts` | `LanguageId`, `LanguageConfig`, `targetLanguageId`, `Phrase.languageId`, `romanization` 추가 |
| `src/data/languages.seed.ts` | 영어, 베트남어, 일본어 개발 순서와 locale 설정 추가 |
| `web/src/lib/language-storage.ts` | 언어별 IndexedDB key 규칙 추가 |
| `web/src/lib/store.ts` | 기본 학습 언어를 영어로 저장하고 변경 action 추가 |

현재 구현은 영어 안정화를 우선하기 위해 UI에서 언어 전환을 노출하지 않는다. 베트남어 콘텐츠 개발 단계에서 언어 전환 UI와 언어별 상태 로딩/전환 UX를 추가한다.
